# Testing Principles（需求驱动）

## 目标

把“需求文档/接口契约/UI 交互”转成可执行、可追踪的测试覆盖，避免只测 happy path。

## 核心规则

1. **需求映射**：每条 `REQ-xxx` / Acceptance Criteria 至少对应 1 个测试用例
2. **分层清晰**：
   - 单元测试：纯函数/领域逻辑/边界分支
   - 集成测试：接口路由、鉴权、DB、外部依赖（可用 mock）
   - E2E：关键链路（尽量少但要稳）
3. **契约优先**：对 API 的 request/response schema、错误码、权限做 contract tests
4. **边界与错误**：空值、最大长度、非法枚举、权限不足、并发/幂等、超时/重试
5. **状态机覆盖**：若有状态（draft→published 等），覆盖所有合法转换与拒绝路径

## 常用覆盖清单

- 输入校验：必填、类型、范围、格式、去空格、重复提交
- 权限：未登录/登录但无权限/资源所有者
- 幂等：重复请求是否安全（尤其是 POST/支付/写操作）
- 竞态：并发更新、乐观锁、重复点击
- 可观测性：关键错误是否记录日志/trace id
